What the product is

Not a chatbot. Not a generic upload-to-GPT page. It is a real service desk workflow with AI inside the process.

📥

Capture work

Website enquiries, email-like messages, manual tickets, and uploaded files all become trackable tickets.

🧠

Understand work

AI classifies intent, urgency, customer, missing information, document fields, risks, and suggested next action.

✅

Control work

Staff review drafts, approve replies, create quotes, schedule follow-ups, and see every AI decision in an audit trail.

End-to-end data flow

Every inbound item is saved first, then processed asynchronously. AI is helpful, but it does not become the source of truth.

Blue = app routes/actionsGreen = AI worker layerYellow = async queuePurple = human review UI

Backend architecture

Keep the app portable: React Router framework mode for the full-stack web app, PostgreSQL as source of truth, Drizzle for schema/query code, pg-boss for reliable jobs, AI SDK isolated behind service functions.

Layer map

1

UI routes

Dashboard pages, ticket detail, inbox, settings, AI monitor, public enquiry widget. Loaders read data. Actions mutate data.

2

Domain services

ticketService, messageService, documentService, quoteService, aiService, queueService, auditService. Routes call services instead of touching everything directly.

3

Persistence

PostgreSQL tables defined with Drizzle. Every raw inbound item is saved before AI runs. AI outputs are stored as suggestions and logs.

4

Workers

Separate Node worker process handles triage, document extraction, reply drafts, quote drafts, reminders, notifications, and eval checks.

pg-boss job design

Use the queue for anything slow, retryable, scheduled, or important to audit.

Core job list

Reliability rules

  • Idempotency key: each job uses ticketId + task + version to avoid duplicate work.
  • Retries: retry AI/network failures; do not retry invalid schema forever.
  • Dead letter: failed jobs are visible in admin UI with “retry” and “mark manual”.
  • Job status reflected in UI: ticket shows “AI triage running”, “document extraction failed”, etc.
  • Small payload: job payload contains IDs, not huge text. Workers fetch fresh data from Postgres.
  • Audit every final action: sending, status change, quote approval, and manual override.

AI service contract

Every AI function should return structured JSON, not only plain text. Validate with Zod before saving business fields.

type TicketTriage = { category: "sales" | "support" | "billing" | "document" | "complaint" | "other"; priority: "low" | "medium" | "high" | "urgent"; summary: string; missingInfo: string[]; suggestedNextAction: string; confidence: number; // 0 to 1 needsHumanReview: boolean; };

Hard AI boundaries

  • AI can draft a reply; it cannot send without approval.
  • AI can suggest quote line items; it cannot finalise price.
  • AI can suggest “resolved”; it cannot close the ticket alone.
  • AI can extract dates and amounts; low-confidence fields require review.
  • AI output must be linked to model, prompt version, source messages, and user approval status.
  • AI should show “why” in short business language, not hidden chain-of-thought.

AI Watchtower: how to keep an eye on AI

This is the part that makes the demo look professional: you are not just adding AI, you are managing AI behaviour.

📊

Run dashboard

Filter by model, task, success/failure, latency, tokens, estimated cost, schema errors, confidence.

🧪

Eval set

Keep 30–50 saved demo cases with expected category, urgency, missing info, and document fields.

🛡️

Safety gates

No external email, final quote, ticket closure, or customer-impacting change without explicit human approval.

📝

Correction loop

When a user edits an AI draft, save the diff. This becomes improvement data and portfolio evidence.

Important product decisions

These decisions keep the app useful, impressive, and buildable.

Use Postgres as the source of truth

Tickets, messages, AI runs, job outcomes, corrections, and approvals should all be inspectable. Avoid hiding important state inside model prompts or temporary memory.

Keep AI behind service functions

Routes and workers call ai.server.ts functions. This lets you swap model/provider, version prompts, run tests, and record every call consistently.

Start with fake email import

A realistic “email inbox” demo can be seeded or webhook-based first. Real Gmail/Outlook integration can come later.

Make approvals the main safety feature

Clients trust systems that show control. Human approval makes the AI useful without being scary.

Make failed jobs visible

A hidden failed AI job looks broken. A visible failed job with retry/manual options looks professional.

Measure edits, not only likes

If staff edit an AI reply heavily, the AI was not good enough. Save that signal in ai_feedback.

Business demo checklist

  • A new enquiry becomes a ticket.
  • AI summary, category, priority, and missing information appear.
  • A draft reply can be edited and approved.
  • A document upload becomes structured extracted fields.
  • A follow-up reminder is scheduled automatically.
  • The ticket timeline clearly shows what happened.

Engineering demo checklist

  • Every AI call is logged in ai_runs.
  • Every background job is visible or auditable.
  • AI output is schema validated before saving.
  • Failed AI/job states are visible in the UI.
  • Prompt versions are stored with outputs.
  • Human approval is required for external actions.
